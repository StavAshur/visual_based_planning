PROJECT OVERVIEW
----------------
Task and Motion Planning (TAMP) for missions requiring a visibility-based device
with a field-of-view (FoV) -- a flashlight or camera -- rather than physical
contact with a target. Target platform: Ubuntu 18.04, ROS Melodic, a Husky
mobile base carrying a UR5 arm with a flashlight (conic FoV).

The material falls into three groups:

  (A) The research paper (LaTeX, in claude_context/) on Target Visibility Motion
      Planning (TVMP): the single-target-set problem solved by two new
      sampling-based planners, VisPRM and VisRRT. Being revised for RA-L.

  (B) Design notes (plain text, in claude_context/) for the next stage,
      Visibility TSP (VisTSP): observing MANY targets with the mobile
      manipulator.

  (C) The ROS package -- this repository. It implements Group A (the TVMP paper:
      VI-tree, VisIK, VisPRM, VisRRT, plus the VIR baseline) and its experiment
      harness. NONE of the Group B / global_plan.txt VisTSP components exist yet;
      see "VisTSP BACKLOG vs. CODE" at the end.

CURRENT STATE IN ONE LINE: the paper's planners are implemented, evaluated, and
running on hardware; the VisTSP extension has not been started in code.

ARCHITECTURAL SHAPE -- read this before diving into files
---------------------------------------------------------
The package is almost entirely HEADER-ONLY. Every algorithm lives in a .h under
include/visual_based_planning/; src/visual_planner.cpp is a 3-line placeholder
that exists only so CMake has a library target. Consequence: when looking for an
implementation, look in include/, not src/. src/ holds only ROS nodes (executables)
and the CUDA kernels.

Runtime topology:
    visual_planning_node  (ROS service server "plan_visibility_path")
        owns -> VisualPlanner
                  |-- ValidityChecker  (MoveIt collision + interpolation)
                  |-- VisualIK         (visibility-aware IK, MoveIt setFromIK)
                  |-- Sampler          (C-space + workspace sampling)
                  |-- PathSmoother     (shortcutting)
                  |-- VisibilityOracle (FoV/occlusion queries, CUDA-backed)
                  |-- VisibilityIntegrity  (the VI-tree -- paper's contribution)
                  |-- VisibilityRoadmap    (clustering variant -- the VIR baseline)
                  |-- GraphManager + NearestNeighbor (roadmap storage + kNN)
    clients: visual_planning_client (single query + execute)
             experiment_runner      (batch benchmark over 5 configurations)

--- include/visual_based_planning/ ------------------------------------------

common/Types.h
  Role: Shared POD types and every default parameter value.
  Significance: The authoritative list of tunables; read it alongside
    planner_config.yaml, since YAML omissions fall back to these defaults.
  Contents: struct Ball {center, radius} (the target minimum-enclosing sphere,
    P_S in the paper); enum EdgeCheckMode {LINEAR, BINARY_SEARCH};
    struct BoundingBox (AABB, min/max per axis -- used everywhere for workspace
    and VI-tree nodes); RRTParams {goal_bias, max_extension, max_iterations};
    PRMParams {num_neighbors, num_samples, max_goals, edge_validation_method,
    max_size}; VisibilityIntegrityParams {num_samples, num_roadmap_samples,
    vi_threshold, k_neighbors, face_samples, limit_diameter_factor};
    VisibilityToolParams {beam_angle (HALF-angle, rad), beam_length}.

VisualPlanner.h  (~950 lines -- the orchestrator; largest file)
  Role: Owns every component and implements both planners end to end.
  Significance: THE entry point for understanding the algorithms. Maps directly
    onto Alg. vis:prm and Alg. vis:rrt in claude_context/4_Method.tex.
  Contents:
    - Constructor: wires ValidityChecker -> VisualIK -> VisibilityOracle ->
      Sampler -> PathSmoother -> robot_state/FK lambda -> target MES ->
      VisibilityIntegrity + VisibilityRoadmap -> initialize_visibility().
    - initialize_visibility(): pulls collision objects out of the MoveIt
      PlanningScene and pushes their AABBs into the oracle, validity checker and
      sampler. Handles shapes::BOX and shapes::SPHERE only (MESH is a TODO), and
      the BOX path ignores shape rotation (comment acknowledges it).
      Then builds the VI-tree or the visibility roadmap if enabled.
    - planVisRRT(): RRT with (a) goal-biased sampling that draws from the VI-tree
      visibility region, (b) per-node direct-visibility goal test via
      checkBallBeamVisibility, (c) a VisIK "snap" branch that converts a
      well-placed POSITION into a target-facing CONFIGURATION. Wall-clock capped
      by time_cap_.
    - planVisPRM(): builds/extends a persistent roadmap ACROSS service calls
      (the graph is not reset between queries -- this is what makes PRM's
      amortized cost low in the paper's results). Scans existing roadmap nodes
      for visibility, samples goals via the VI-tree, expands, and returns a
      Dijkstra path. The no-start overload samples a random valid start while
      rejecting the double-room interior box (0<x<2.25, |y|<2.25).
    - sampleVisibilityGoal(): the shared bridge -- ask VI-tree (or roadmap) for a
      point that sees the target ball, build a look-at pose, solve IK.
    - connectToGraph(), addState(), solveFK(), finalizePath() (shortcut + a final
      collision re-validation of the raw path that logs ROS_ERROR on failure).
  Gotchas: group_name_ defaults to "whole_robot" (9 DOF: 2 planar base + arm)
    with "manipulator" commented out throughout; workspace_bounds_ is also
    hardcoded in the constructor before config overrides it.

components/ValidityChecker.h
  Role: All collision and metric primitives.
  Significance: Every state/edge check in the system funnels through here, so it
    dominates runtime.
  Contents: isValid() (MoveIt checkCollision on the "whole_robot" group -- NOTE
    the group name is hardcoded in this call, not taken from group_name_);
    interpolate() and distance() delegating to MoveIt JointModel so continuous
    vs. revolute vs. prismatic joints are handled correctly; validateEdge() with
    LINEAR and recursive BINARY_SEARCH modes; extend() (RRT steering, returns the
    last valid state before collision); isPointInObstacle() / isPointInWorkspace()
    over Boost.Geometry AABBs. The bottom half of the file is a large
    commented-out earlier version that logged every checked configuration.

components/Sampler.h
  Role: All sampling -- both C-space and workspace.
  Contents: sampleUniform() (MoveIt setToRandomPositions on the group);
    sampleValidPoints() (uniform workspace points, plus optional "face sampling"
    that seeds points just outside each obstacle face by 5 mm -- used by the
    visibility roadmap); sampleInBox() (collision-free points in an AABB, the
    VI-tree's S_in); sampleOutside() (points in B but outside B', optionally
    limited to a `range` band around B' -- the VI-tree's S_out, where range is
    the beam length); computeLookAtRotation() (builds the camera frame whose +Z
    points at the target); two sampleVisibilityAwareState() overloads (ball
    target and point-set target) that grow the sampling radius geometrically
    until an IK-feasible, sufficiently-visible pose is found.

components/VisualIK.h
  Role: Visibility-aware inverse kinematics -- the paper's VisIK
    (4_Method.tex, sec:method:vis:ik).
  Significance: The component that turns "this place can see the target" into
    "this joint configuration sees the target".
  Contents: solveIK() (plain MoveIt setFromIK with a ValidityChecker callback);
    solveVisualIK(current_joints, Ball, orientation, out) -- the core: from beam
    half-angle theta and length h it computes the feasible standoff segment
    [a,b] along the view axis (near bound r/tan(theta), far bound h-r), clips
    that segment against a reachability AABB via the slab method, projects the
    current EE position onto it, and then does a bidirectional local search
    along the segment calling IK at each step; overloads for a point list;
    maintainOrientationVisualIK() (keep current EE orientation);
    greedyOrientationVisualIK() (recompute a look-at orientation first).
  Known issues, marked in-file with @todo: the search direction/step-size logic
    is flagged WRONG by the author ("dir should be equal to v, step size should
    differ per direction"), and there is a commented-out hard-coded flashlight
    offset. The clipping AABB (-5,-5,0)..(5,5,1.4) is hardcoded here.

components/PathSmoother.h
  Role: Path post-processing (the `enable_shortcutting` option).
  Contents: smoothPath() -- exact shortcutting by dynamic programming: treats the
    waypoints as a DAG, edge i->j exists if validateEdge succeeds, and takes the
    minimum-cost chain. Not randomized shortcutting; it is optimal over the given
    waypoint set but costs O(n^2) edge checks.

components/PathExecuter.h
  Role: Sends a joint-space path to the real/simulated robot.
  Significance: The only file that talks to MoveGroupInterface for execution;
    used by visual_planning_client, not by the planner service.
  Contents: builds a RobotTrajectory, runs IterativeParabolicTimeParameterization
    with velocity/acceleration scaling (defaults 0.025 / 0.05 -- deliberately
    slow), then execute(). A `wait_time > 0` debug mode instead moves waypoint by
    waypoint with a 2 s pause between each (useful for watching the FoV).

data_structures/VisibilityIntegrity.h  (the VI-TREE -- key contribution)
  Role: Implements the Visibility Integrity tree from 4_Method.tex
    (sec:method:visibility:integrity:tree; Algs. vi:tree, vi:tree:recursive,
    vi:tree:query).
  Significance: This is the paper's headline data structure. It answers "which
    regions of the workspace can see this target?" without per-query raycasting.
  Contents:
    - struct VINode: AABB, left/right/parent, is_leaf, height, path_idx string
      (e.g. "0101" for debugging), convexity_score, and
      visible_from_nodes: vector<(VINode*, score)> -- the inverse-visibility list.
    - build(): recursive AABB partition, then ComputeLeafPairwiseVisibility(),
      then ComputeParentVisibilityBottomUp().
    - buildTreeRecursive(): samples S_in inside the box (count scaled down with
      box volume) and S_out in the environment around it (within beam_length),
      uploads the combined set to the GPU as one NxN visibility matrix
      (precomputeGpuMatrix), then computes the VI score as
      |{S_out seen by ALL of S_in}| / |{S_out seen by ANY of S_in}| using the
      checkSeenByAll/Any reduction kernels. Splits on the longest axis while the
      score is below vi_threshold; makes a leaf when the score passes or the box
      shrinks below 0.1 m. Also derives a convexity_score from the same data.
    - ComputeLeafPairwiseVisibility(): one all-pairs GPU batch over leaf centers.
    - ComputeParentVisibilityBottomUp(): a parent's visibility list is the
      INTERSECTION of its children's (sorted-merge), with the score averaged, and
      the relation is written back into the seer's list so queries at any depth
      work. Single-child nodes copy and propagate.
    - query(Eigen::Vector3d, out): descends to the containing leaf, accumulating
      visible_from_nodes from every ancestor on the path.
    - query(Ball, out): the version actually used -- traverses, prunes boxes
      outside the ball, and accumulates a per-seer BALL FRACTION. The covering
      nodes (fully-contained nodes, where recursion stops, plus partially
      intersecting leaves) form a disjoint cover of the ball, so their fractions
      can be summed: each seer's score is the dimensionless fraction of the target
      ball visible from it, in [0,1].
    - SampleFromVisibilityRegion(Ball, out_point, threshold): rejection sampling
      -- pick a seer, accept it with probability (its ball fraction / the max ball
      fraction over all seers), sample a collision-free point in its box, and
      confirm with a real checkBallBeamVisibility call. The rescale by the max is
      deliberate: raw fractions are usually well below 1 (a seer rarely covers a
      whole target), so using them directly would reject nearly every candidate.
      Dividing by the max preserves the relative ranking -- all that rejection
      sampling needs -- while keeping the best candidate's acceptance rate at 1.
    - Geometry helpers: distSqPointBox, distSqBoxBox, isBoxFullyInBall,
      getBoxVolume/Size, getBallVolume, getIntersectionVolume (raw m^3 of
      box n ball; exact when the box is fully inside, Monte-Carlo otherwise),
      getBallFraction (= getIntersectionVolume / getBallVolume). getBallFraction
      is the SINGLE normalization point -- both the fully-contained and the
      partial case go through it, so scores stay comparable across nodes.
  Gotchas: makeLeaf() APPENDS every leaf to "leaf_boxes.txt" in the process CWD
    (usually ~/.ros) on every build -- a debug/visualization dump that grows
    across runs. intersectsWorkspaceBox() hardcodes the reachable goal AABB
    (-5,-5,0)..(5,5,1.4); intersectsWorkspaceSphere() is an unused alternative
    with a hardcoded 1.2 m sphere at (0.33, 0, 0.33).

data_structures/VisibilityRoadmap.h
  Role: The ALTERNATIVE to the VI-tree: a clustering-based visibility structure.
    Selected by use_visibility_roadmap:=true. This is the "VIR" configuration in
    experiment_runner -- i.e. the Visibility Roadmap BASELINE from
    2_Related_work.tex, not a contribution.
  Significance: Same public query surface as VisibilityIntegrity
    (SampleFromVisibilityRegion, query, isConnectedToTarget), so VisualPlanner
    can swap one for the other behind a single boolean.
  Contents: build() = sample N valid workspace points (with optional face
    sampling) -> computeSampleVisibility (all-pairs, GPU with a CPU fallback,
    stored as sorted adjacency lists) -> performClustering() ->
    buildClusterVisibilityGraph() -> build a kNN index over the samples.
    Clustering is greedy: seed on a random point, walk outward by distance, admit
    a point while the incremental VI score (|intersection of visibility sets| /
    |union|) stays >= vi_threshold, stop when the diameter exceeds
    limit_diameter_factor * current diameter. Each cluster gets a centroid and a
    bounding radius. The cluster graph gets an edge when random member pairs are
    mutually visible, weighted by the observed fraction, and is pruned to
    clusters intersecting the goal AABB. query() classifies a point by kNN
    majority vote over cluster labels. Prints extensive cluster statistics and
    dumps "clustered_points.txt".
  Gotcha: goal AABB (-5,-5,0)..(5,5,1.4) is hardcoded again here.

data_structures/VisibilityOracle.h
  Role: All visibility/FoV queries. The CPU-side API over the CUDA kernels.
  Significance: The performance-critical core; every planner decision is
    ultimately an oracle call.
  Contents:
    - Obstacle store as Boost.Geometry Box3D, mirrored to GPU lazily via a
      gpu_dirty_ flag (updateGPUObstacles).
    - isPointInFOV(): the cone test -- transform the target into the sensor
      frame, require +Z half-space, distance <= beam_length, and
      cos(angle) >= cos(beam_angle).
    - isPointInBeam() = isPointInFOV && unoccluded segment.
    - checkBallBeamVisibility(): the central scoring function -- Monte-Carlo
      samples num_samples_ (100) points in the target ball and returns the
      illuminated fraction. Overloads take an Isometry3d pose, a joint vector
      (via the injected FK lambda), or a bare source position (in which case a
      look-at frame is synthesized). NOTE: the GPU batch branch inside this
      function is disabled by `if (false)` -- it always runs the CPU path.
    - computeBatchVisibility(): all-pairs NxN on GPU.
    - checkBatchVisibility(): a list of specific segments on GPU.
    - precomputeGpuMatrix() + checkSeenByAllGPU() / checkSeenByAnyGPU(): upload
      an NxN matrix once and run cheap index-only reductions against it -- this
      is what makes VI-tree construction tractable.
    - VisibilityMethod enum {BF, GPUBF, KDTree}; default GPUBF; KDTree throws.

data_structures/VisibilityOracleGPU.h
  Role: The C/CUDA boundary -- POD structs and kernel launcher declarations, free
    of Boost/Eigen so the .cu file can include it.
  Contents: struct GPUBox, struct GPUPoint; declarations of
    launchVisibilityKernel, launchAllPairsVisibilityKernel,
    launchBatchSegmentKernel, launchComputeSeenByAll, launchComputeSeenByAny.

data_structures/Graph.h
  Role: Roadmap/tree storage and graph algorithms (namespace-free; class
    GraphManager).
  Contents: Boost adjacency_list<vecS, vecS, undirectedS> with GraphVertex
    {joint_config, ee_pose} and GraphEdge {weight}; addVertex/addEdge (note:
    addEdge inserts BOTH directions into an already-undirected graph, so each
    logical edge appears twice); getVertexConfig/getVertexPose; isEdge;
    shortestPath (Boost Dijkstra + backtracking); inSameComponent (BFS with early
    exit -- used as a cheap pre-check before Dijkstra); countConnectedComponents
    (prints the k largest with a representative config); exportCCMap() (writes
    cc_map.txt: id, joint values, connected-component id -- for offline analysis
    of why PRM failed to connect); printVertexDegrees.

data_structures/NearestNeighbor.h
  Role: kNN over configurations, with a robot-aware metric.
  Significance: Contains a real performance trap -- see below.
  Contents: PointCloud adaptor; RobotStateDistance metric that uses MoveIt
    JointModel::distance per joint and then WEIGHTS the dimensions -- prismatic
    joints (the Husky base translations) by the DOF count, revolute joints by
    (n - i) so proximal arm joints dominate distal ones. This weighting is what
    makes nearest-neighbour queries prefer moving the arm over driving the base.
    Wraps nanoflann KDTreeSingleIndexAdaptor; also provides bruteForceKNN.
  Gotcha: addPoint() DELETES AND REBUILDS the entire kd-tree on every insertion,
    making roadmap construction O(n^2 log n). This is the most likely target if
    PRM build time needs to come down (cf. commit "trying to make NN faster").

data_structures/nanoflann.hpp
  Role: Vendored third-party header (nanoflann kd-tree). Do not edit.


--- src/ --------------------------------------------------------------------

src/visual_planning_node.cpp
  Role: The main ROS service node -- the production entry point.
  Contents: class VisualPlanningNode wraps a PlanningSceneMonitor and a
    VisualPlanner and advertises "plan_visibility_path".
    - loadStaticConfig(): reads every parameter from the PRIVATE namespace
      (planner_config.yaml) into the planner.
    - updatePlannerParams(): on EVERY service call, re-reads five switches from
      the GLOBAL namespace (/planner/use_visual_ik, /planner/use_visibility_integrity,
      /planner/use_visibility_roadmap, /planner/enable_shortcutting, planner/mode)
      and sets params_changed_ if any differ. This is the mechanism
      experiment_runner uses to switch algorithms without restarting the node.
    - planCallback(): if uninitialized OR params changed, resets the planner and
      re-snapshots the planning scene (this is what triggers VI-tree rebuild);
      computes the target MES from req.task.target_points; takes start_joints from
      the request or the current state; dispatches to planVisPRM if the mode
      string contains "PRM", else planVisRRT; packs the result into a
      JointTrajectory with the group's active joint names.
    - main(): RobotModelLoader + PlanningSceneMonitor with scene/world/state
      monitors, AsyncSpinner(2).

src/visual_planning_client.cpp
  Role: Single-query client: request a plan and execute it on the robot.
  Contents: resolves the planner mode (private ~mode -> /visual_planning_node/
    planner/mode -> "VisRRT" default); reads the flat /target_points array
    (published by the ZED tracker, see readme) into Point[]; calls the service;
    converts the returned trajectory back to vector<vector<double>> and runs it
    through PathExecuter("whole_robot") at scaling 1.0/1.0.

src/experiment_runner.cpp
  Role: The benchmark harness that produced the paper's numbers.
  Significance: The single best file for understanding the experimental protocol
    in claude_context/5_Experiments.tex.
  Contents:
    - Five configurations compared, each a (mode, use_visual_ik,
      use_visibility_integrity, use_visibility_roadmap) tuple:
        VisRRT (VisRRT + VisIK), RRT (plain), VisPRM (PRM + VI-tree),
        VIR (PRM + visibility roadmap = the baseline), PRM (plain).
      Switched at runtime by setting the global /planner/* params.
    - generateTargetProblemsFromRegions(): loads sampling_regions_double_room.yaml,
      pulls the real obstacle AABBs out of the live MoveIt planning scene, and
      pre-generates 200 target problems -- each a sphere sampled inside a region
      with the largest radius that fits, rejected if it intersects any obstacle,
      then populated with 100 uniformly-distributed interior points.
    - 200 trials per configuration; trial 0 is recorded separately as the
      initialization/build cost (the amortization argument in the paper).
    - computeStats (avg/median/min/max/stddev), printStats, saveToFile ->
      planner_experiments_results.txt, including per-run success flags and a
      per-target room flag (x>0 = "windows" room, x<0 = "door" room).
  Bug worth knowing: room_flags_ is only `reserve`d, never `resize`d, but is
    written via `room_flags_[i] = ...` -- out-of-bounds writes to a vector of
    size 0. The room column in saved results is therefore unreliable.

src/VisibilityOracleGPU.cu
  Role: All CUDA kernels + their host launchers.
  Contents:
    - __device__ checkSlabIntersection(): ray/AABB slab test, the primitive
      underneath every visibility query.
    - checkIntersectionKernel: one segment vs. all obstacles, early-outs on a
      shared found flag.
    - allPairsVisibilityKernel: NxN point-pair visibility (fills the matrix used
      by the VI-tree and the visibility roadmap).
    - batchSegmentKernel: M independent segments.
    - seenByAllKernel / seenByAnyKernel: reductions over a cached NxN matrix
      against a guard-index set -- the numerator and denominator of the VI score.
    - Host wrappers matching VisibilityOracleGPU.h.

src/visual_ik_server.cpp

CROSS-CUTTING NOTES (things that will bite you)
-----------------------------------------------
1. "whole_robot" vs "manipulator": the planning group was switched from the UR5
   alone to the full 9-DOF mobile manipulator. The old value survives as
   commented-out lines in VisualPlanner.h, ValidityChecker.h, NearestNeighbor.h,
   visual_ik_server.cpp, visual_ik_client.cpp, visual_planning_client.cpp.
   ValidityChecker::isValid() still hardcodes "whole_robot" in its
   getJointModelGroup call rather than using group_name_.
2. The reachable/goal AABB (-5,-5,0)..(5,5,1.4) is hardcoded in THREE places:
   VisibilityIntegrity::intersectsWorkspaceBox, VisibilityRoadmap::
   buildClusterVisibilityGraph, and VisualIK::solveVisualIK. Changing the
   environment means changing all three.
3. Debug artifacts are written to the process CWD (~/.ros when launched by
   roslaunch): leaf_boxes.txt (appended, grows unboundedly), clustered_points.txt,
   cc_map.txt.
4. The CUDA arch flag is pinned to sm_89 in CMakeLists.txt.
5. VisibilityOracle::checkBallBeamVisibility's GPU branch is dead code
   (`if (false)`); ball visibility scoring is CPU-bound.
6. The PRM roadmap deliberately PERSISTS across service calls; only a param
   change or scene change resets it. Timing numbers only make sense in that light.

VisTSP_problem.txt
  Role: Problem statement for Visibility TSP and its subproblems.
  Contents: VisTSP = find a motion plan whose swept FoV covers a target set P.
    Four subproblems: (1) regions such that each target is seen by >= k regions
    (VI-tree multi-target query + pruning); (2) 2D floor locations for the base
    whose workspace spheres cover those regions (optional beta coverage
    fraction); (3) a local plan seeing a location's targets with the base fixed
    within delta; (4) a global plan routing the base through a covering subset.

  Role: Backlog of components still needed. Maps 1:1 onto global_plan.txt.
  Contents: (1) VI-tree multi-target query with coverage parameters (vector of
    target spheres, beta fraction, returns region->targets map); (2) region
    pruning (keep every target covered >= k times while shrinking the region
    set); (3) set-cover of workspace regions by reachable-volume copies;
    (4) local TSP (3D, base ~fixed); (5) global TSP (2D, arm ~irrelevant).

VisTSP_frameworks.txt
  Role: Two candidate end-to-end frameworks.
  Contents: GLVisTSP (preferred): regions -> base locations covering them ->
    global TSP over locations -> per-location local VisTSP -> execute.
    VisCfgTSP (benchmark): configs such that each target is watched by >= K
    configs -> TSP visiting one watching config per target.

local_VisTSP.txt
  Role: Detailed sketch of the LOCAL VisTSP algorithm (subproblem 3).
  Contents: Local VisRRG(r, E, B, T, k) -- sample the VI-tree for target configs,
    connect them, extend toward under-observed targets using visibility checks +
    VisIK, all within a local bounding box B; "TSP Greek Alg" -- MST -> post-order
    traversal -> shortcut with a local planner, falling back to RRT/VisRRT when a
    shortcut is hard (the interesting case); LocalVisTSP = Local VisRRG then
    GreekTSP.

HOW THE PIECES RELATE
---------------------
The paper (Part 2, Group A) defines the primitives -- the VI measure, the VI-tree,
VisIK, and the VisPRM/VisRRT planners for single-goal TVMP. Part 1 (the repo) is
their implementation on the Husky + UR5, together with the VIR baseline and the
benchmark harness that produced 5_Experiments.tex. The VisTSP notes (Part 2,
Group B) reuse those same primitives -- above all the VI-tree query and VisIK --
to build multi-target coverage: regions from the VI-tree, base locations covering
those regions, a global tour over locations, and local visibility tours at each
stop. global_plan.txt sequences that into 6 build steps, none of which are
implemented yet (Part 3).
